# Week 5 - What Is Due This Week

## Read this first
You have **2 separate things** to keep straight this week:

1. **Week 5 Sprint Packet**  
   This is the **weekly team submission** for this sprint.

2. **Sprint 1 Document**  
   This is a **separate project document**. It is **not** the same thing as the Week 5 Sprint Packet.

Do not combine them.

---

## Part A: Week 5 Sprint Packet

### Main goal for this week
Your team must ship **one small vertical slice** through a real workflow:

**Issue → Branch → PR → Review → Check/CI → Demo or Deploy**

A **vertical slice** means one small feature that works end-to-end. Examples:
- one form submits
- one page loads data
- one action saves correctly
- one user flow can be demonstrated

These do **not** count:
- “worked on backend”
- “did UI”
- “fixed stuff”
- “made progress”

---

## What your team must have by the end of Week 5 work

### In GitHub
Your team should have:
- **1 Issue** for the vertical slice
- **1 branch**
- **1 Pull Request (PR)**
- **1 review comment**
- **1 check / CI result**
- **1 demo link or deploy link**

### Quality rules
Your Week 5 work should follow these rules:
- Work should start from an **Issue**
- Every active task should have **one owner**
- Every active task should have a **Definition of Done (DoD)**
- The PR should explain:
  - **What changed**
  - **Why**
  - **How tested**
  - **Evidence**
- Do **not** work directly on `main`
- Do **not** make one huge PR with mixed work

---

## What must be inside the Week 5 Sprint Packet

### 1) Demo
Include:
- **Demo link** (live link or video)
- **3-bullet demo script**
- **Backup plan** if live demo fails

### 2) What you shipped
List the most important things your team shipped this week.
Each item must have a **real link**.

Good links:
- PR link
- Issue link
- review comment link
- CI/check link
- screenshot or short video

### 3) Board snapshot
Show your team board with:
- **To Do**
- **Doing**
- **Done**

Every active card must have:
- **one owner**
- **Definition of Done**

If something is blocked, label it **blocker** and explain what is needed.

### 4) Engineering practice spine
For this sprint, show evidence for **at least one** of these:
- Testing basics
- CI checks
- Deployment reliability
- Logging / observability
- Security basics
- Accessibility basics
- Performance basics

For Week 5, the most natural choice is:
- **CI checks**
- **Deployment reliability**

### 5) What broke / Bugs
Add bug links if something failed or broke.

### 6) Risks / Blockers
List:
- blocker
- what you need
- owner
- link

### 7) Plan for next sprint
Add:
- top 3 next goals
- scope cut list (what gets cut first if time runs out)

### 8) Individual contribution receipts
Each student must comment on the Sprint Packet with:
- **2–3 links minimum**
- short 1-sentence summary of contribution

What counts:
- PR link
- Issue link you owned and completed
- review comment link
- CI run link
- screenshot attached to Issue or PR

What does **not** count:
- “I worked on it”
- “we discussed it”
- “almost done”

## Part B: Sprint 1 Document (separate from Week 5 Sprint Packet)

## Important
The **Sprint 1 Document** is a separate document.  
It is **not** the Week 5 Sprint Packet.

The Sprint 1 Document should show the foundation of your team’s first real build plan.

---

## What should be in the Sprint 1 Document

### 1) Sprint goal
Write **one clear sprint goal**.

Bad:
- work on app
- improve system
- do frontend

Better:
- user can sign up with email
- user can submit reservation form
- admin can view one list of submitted entries

### 2) 1–3 user stories maximum
Keep it tight.
Write only the most important user stories for the sprint.

### 3) One chosen story for the sprint
Make clear:
- which story you chose first
- why you chose it

### 4) Buildable GitHub Issues
Turn the chosen story into a **real Issue**.
A good Issue should include:
- **What**
- **Why**
- **Owner**
- **Definition of Done**

### 5) Definition of Done for active work
Every active issue should have a DoD that can be checked.

Example:
- [ ] form loads
- [ ] valid input works
- [ ] invalid input shows message
- [ ] screenshot attached
- [ ] owner assigned

### 6) Ownership
Every active issue should have **one owner**.

### 7) First slice started
The document should show that work moved beyond planning.
Include proof that the first slice actually started:
- branch created
- work started
- at least one commit
- linked Issue or PR

### 8) Evidence links
The document should include links or references to proof, not vague claims.

### 9) Risks or blockers
If something is slowing the team down, name it clearly.

### 10) Next move
State the next build step clearly.

---

## What “good” looks like for the Sprint 1 Document
A good Sprint 1 Document shows:
- one clear sprint goal
- 1–3 user stories only
- one chosen story turned into a real Issue
- owner on every active issue
- DoD on every active issue
- branch created
- work started
- proof links included
- next move is obvious

---

## Common mistakes to avoid

### Week 5 mistakes
- working on `main`
- no Issue first
- huge PRs
- no “How tested”
- no review comment
- no CI/check evidence
- no demo/deploy link
- no receipts

### Sprint 1 Document mistakes
- vague sprint goal
- too many user stories
- no owner
- no DoD
- no proof links
- planning only, no build started
- mixing Sprint 1 Document content into the Week 5 Sprint Packet
